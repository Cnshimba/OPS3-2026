The Storage Abstraction Layer

Proxmox abstracts physical hardware into "Storage Pools".

Layer Example
Physical SSD, NVMe, HDD
Logical LVM, ZFS, RAID Controller
Storage Backend Directory, NFS, iSCSI, Ceph
Virtual Disk .qcow2, .raw, .vmdk
---

Storage Models: Local

Storage that lives inside the server case.

  • Directory (Dir):
    • Simple file system (ext4/xfs).
    • Stores images as files.
  • LVM-Thin:
    • Block storage.
    • Allows "Thin Provisioning" (100GB disk uses 0GB initially).
  • ZFS (Local):
    • Enterprise grade.
    • Features: Compression, Deduplication, Bit-rot protection.

Storage Models: Shared

Storage accessible by multiple nodes over the network.

  • NFS (Network File System):
    • File-level sharing (Like a shared folder).
    • Best for: ISOs, Backups.
  • iSCSI:
    • Block-level sharing.
    • Best for: High-performance databases.
  • Ceph:
    • Distributed Storage.
    • Pools local disks from 3+ servers into a virtual SAN.
    • Self-healing and highly scalable.

Thick vs. Thin Provisioning

Feature Thick Provisioning Thin Provisioning
Allocation Reserves 100% space upfront. Reserves 0% upfront; grows on write.
Performance Faster (No allocation overhead). Slightly slower.
Safety High (Guaranteed space). Risk of "Overprovisioning".
Use Case Critical Databases. Lab VMs, Web Servers.
---

ZFS: The Zettabyte File System

More than just a file system.

  • Copy-on-Write (CoW): Never overwrites data. Always writes to new blocks.
  • Instant Snapshots: Adding a bookmark takes 0 seconds.
  • Data Integrity: Checksums every block. Automatically fixes "Bit Rot".

"ZFS is the last word in filesystems."

The Golden Rule: Snapshot != Backup

Snapshot

  • State: Dependent on the original disk.
  • Storage: Same drive as the VM.
  • Survives Disk Failure? NO.
  • Use: Before a Windows Update.

Backup

  • State: Independent full copy.
  • Storage: External drive / NAS / Cloud.
  • Survives Disk Failure? YES.
  • Use: Disaster Recovery.

Proxmox Backup Modes

  1. Stop Mode:

    • VM is powered off.
    • Pros: 100% Consistency.
    • Cons: Downtime.

    • Suspend Mode:

    • VM is frozen.

    • Pros: Minimal downtime.

    • Snapshot Mode (Live):

    • VM keeps running.

    • Pros: Zero Downtime.
    • Note: QEMU handles the complexity in the background.

Lab Overview

  1. Explore Storage: Use lsblk and df -h.
  2. LVM Mastery: Create volumes manually.

  3. Disaster Sim:

    • Create a file.
    • Snapshot.
    • Delete the file.
    • ROLLBACK and save the day.
    • Backup: Run a full VZDump backup.

Lab 2: Shared Storage Simulation

Task: Build a Virtual SAN

  1. The Server: Configure a Linux VM to act as an NFS Server (/etc/exports).
  2. The Client: Connect Proxmox to this NFS share via Datacenter > Storage.

  3. The Test:

    • Store an ISO on the NFS share.
    • Delete the NFS VM... and watch the connection drop (Simulated Failure).
    • Restore the VM... and watch Proxmox reconnect (Self-Healing).
