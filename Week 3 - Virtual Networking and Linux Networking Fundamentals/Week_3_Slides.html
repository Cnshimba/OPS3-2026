Virtual Networking and Linux Networking Fundamentals

Week 3

Operating Systems 3 (Virtualisation & Cloud Technologies) Instructor: KT Nshimba

Welcome to Week 3!

Networking is the nervous system of specific virtualized infrastructure.

  • Traditional: Physical cables, hardware switches.
  • Virtual/Cloud: Software-defined wires, bridges, and routers.

The Goal Bridge the gap between "clicking a button in Proxmox" and understanding the kernel-level mechanics (ip, bridge, iptables) that make that click possible.

Part 1: Linux Networking Fundamentals

Proxmox is Debian Linux. To manage it, you must master Linux networking.

1. Runtime vs. Persistent Networking

Feature Runtime (ip) Persistent (nmcli / Config)
Layer Kernel Layer (Netlink) Configuration Daemon
Speed Instant (Microseconds) Applies on reload/reboot
Scope In-Memory (RAM) Written to Disk (/etc/...)
Risk LOST ON REBOOT SAVED
Best Practice:
Use ip for instantaneous troubleshooting. Use nmcli or /etc/network/interfaces for permanent changes.

1.1 The Kernel Layer: ip (Runtime)

The ip command is part of the modern iproute2 suite.

  • Link Status: ip link show
    • Shows Up/Down state, MTU, and MAC addresses.
  • IP Addresses: ip addr show
    • Shows IPv4 (inet) and IPv6 (inet6) addresses attached to interfaces.
  • Routing Table: ip route get 8.8.8.8
    • Simulates a packet path to tell you which interface and gateway would be used.

1.2 The Configuration Layer: nmcli (Persistent)

  • Profiles: Settings generally saved in /etc/NetworkManager/system-connections/.
  • Logic: Handles complex logic like "Connect to WiFi if Ethernet fails."
  • Example Command: bash # Set IP and Gateway persistently nmcli con mod eno1 ipv4.addresses 192.168.1.50/24 nmcli con mod eno1 ipv4.gateway 192.168.1.1 # Apply changes nmcli con up eno1

Part 2: Network Namespaces

2.1 The "Tenant Problem"

Scenario: Imagine you are hosting Amazon Web Services.

  • Customer A wants a private network using 192.168.1.0/24.
  • Customer B also wants to use 192.168.1.0/24.

The Conflict: On a normal OS, you cannot have the same IP address twice. It would cause an IP conflict.

2.2 The Solution: Namespaces

A Network Namespace is a fundamental Linux kernel feature. It provides a completely encapsulated network stack:

  • Interfaces: Its own eth0, lo.
  • Routes: Its own Routing Table.
  • Firewall: Its own iptables chains.

Result: Customer A lives in Namespace A. Customer B lives in Namespace B. They use identical IPs without ever conflicting. This is the technology behind Docker and Kubernetes.

Part 3: Connecting the Dots

3.1 The Virtual Cable (veth)

Isolation is safe, but useless if we cannot communicate.

  • Definition: Virtual Ethernet pair. A "pipe" with two ends.
  • Behavior: Packets sent into Interface A instantly arrive at Interface B.
  • Usage:
    • Plug one end into the Namespace ("inside").
    • Plug the other end into the Host ("outside").

3.2 The Virtual Switch (bridge)

  • Definition: A software implementation of a Layer 2 Ethernet Switch.
  • Function: It maintains a MAC Address Table and forwards frames.
  • Proxmox Context:
    • vmbr0 acts as the physical switch.
    • Physical NIC (eno1) is the uplink.
    • VM Interfaces (tap100i0) are the downlinks.

Note on VMs: VMs use tap interfaces (Simulated L2 Ethernet) to plug into bridges.

Part 4: Container Networking

4.1 Docker (The Standard Bridge)

  • Architecture: Docker creates a bridge named docker0 by default.
  • IPAM: It manages its own internal IP pool (172.17.0.x).
  • Connectivity:
    • Every container gets a veth pair.
    • Traffic to the outside world is NAT'd (Masqueraded) via the host IP.

4.2 Podman and Rootless Networking

  • Security Focus: Podman can run containers as a standard user (non-root).
  • Challenge: Standard users cannot create Global Network Interfaces.
  • Solution: slirp4netns
    • Uses a user-space network stack to translate packets.
    • Slower than Kernel bridging, but much more secure.

4.3 LXC (System Containers)

  • Approach: LXC containers behave like lightweight Virtual Machines.
  • Network:
    • Often bridged directly to the physical LAN (vmbr0).
    • Requests an IP from your physical Router via DHCP.
    • Visible to the rest of the network as a distinct device.

Part 5: Advanced Linux Operations

5.1 Link Aggregation (Bonding)

  • Problem: Single Point of Failure. If eno1 breaks, the server goes offline.
  • Solution: Bond eno1 + eno2 -> bond0.
  • Modes:
    • Active-Backup: One cable works. If it fails, the other takes over instantly (Default).
    • LACP (Mode 4): Uses both cables for Speed + Redundancy. Requires Switch Support (802.3ad).

5.2 IP Forwarding (Routing)

  • Host Behavior: By default, Linux acts as a specific destination. It drops packets not meant for its own IP.
  • Router Behavior: To act as a Gateway (passing packets from VM -> Internet), we must enable forwarding.

Command:

sysctl -w net.ipv4.ip_forward=1

5.3 The Linux Firewall (iptables)

The "Security Group" of the kernel.

Chains: 1. INPUT: Traffic TO the host (e.g., SSH). 2. OUTPUT: Traffic FROM the host. 3. FORWARD: Traffic THROUGH the host (VMs).

Frontends:

  • Firewalld (RHEL): Zone-based management.
  • UFW (Ubuntu): Profile-based management ("Allow SSH").

5.4 Network Reconnaissance

Tools for Discovery:

  1. Socket Statistics (ss)

    • Inspect local listening ports.
    • ss -tulpn -> "Is port 80 actually open?"

  2. Network Mapper (nmap)

    • Scan remote targets.
    • nmap 192.168.1.50 -> "What ports is that server hiding?"

Part 6: Open vSwitch (OVS)

6.1 Architecture

The Cloud-Grade Switch.

  • Control Plane: ovs-vswitchd. Handles complex logic, High Availability, and API calls.
  • Data Plane: openvswitch.ko. Kernel module that switches packets at wire speed.

6.2 Key Features

Why use OVS over Linux Bridge?

  1. SDN: Fully programmable via centralized controllers (OpenStack Neutron).
  2. Overlay Networks: VXLAN and GRE tunneling support for multi-tenant clouds.
  3. Telemetry: sFlow/NetFlow support to analyze every single packet flow.

Part 7: Proxmox Networking

7.1 Networking Modes

1. Bridged Mode (Default)

  • VM is a full citizen on the LAN.
  • Traffic flows: VM -> Bridge -> Physical Router.

2. NAT Mode

  • VM sits behind the Host's IP.
  • Secure, but requires Port Forwarding for inbound access.

3. VLAN Tagging

  • Proxmox tags packets leaving the VM interface (e.g., Tag 10).
  • Enables segmentation (Student VLAN vs Admin VLAN).

Summary

  • Tools: Move from ifconfig to ip.
  • Isolation: Namespaces are the foundation of Containers.
  • Connectivity: Bridges + Veth/Tap cables.
  • Advanced: Bonding for redundancy, OVS for SDN.
