Week 11 Lab 4: Container Orchestration with Magnum

Module: Operating Systems 3 (Virtualisation & Cloud Technologies)

Estimated Time: 60 Minutes
Lab Type: CLI / Kubernetes

Lab Overview

In Lab 3, you used Heat to deploy Virtual Machines. Now, we will use OpenStack Magnum to deploy a Kubernetes Cluster. Magnum uses Heat under the hood to build the nodes, but once it's done, we will switch tools to kubectl to deploy Containers.

Objectives: 1. Create a Cluster Template: Define the OS and Docker engine settings. 2. Launch a Cluster: Provision 3 VMs (1 Master, 2 Workers) automatically. 3. Configure Access: Connect your local kubectl client to the remote cluster. 4. Deploy an App: Launch Nginx with 3 replicas and a Load Balancer.

Prerequisites:

  • MicroStack running.
  • Patience: Creating a cluster involves launching multiple VMs and installing Kubernetes. It takes approx. 15-20 minutes.

Part 1: The Blueprint (Cluster Template)

  1. Create the Template: We need to tell Magnum what kind of OS to use. Kubernetes requires a specialized "Atomic" OS, usually Fedora CoreOS.
openstack coe cluster template create k8s-lab-template \
  --image fedora-coreos-35 \
  --keypair lab-key \
  --external-network external \
  --dns-nameserver 8.8.8.8 \
  --flavor m1.medium \
  --master-flavor m1.medium \
  --coe kubernetes

*   `--coe`: Container Orchestration Engine (Kubernetes is the standard).
*   `--image`: This image must exist in Glance (your instructor should have uploaded it).
  1. Verify: bash openstack coe cluster template list

Part 2: Warning - The Wait

We are about to launch the cluster. Once you run this command, do not interrupt it. It will take time.

  1. Launch: bash openstack coe cluster create k8s-cluster \ --template k8s-lab-template \ --master-count 1 \ --node-count 2

    • 1 Master: Controls the cluster (API Server).
    • 2 Nodes: Where your apps run.

  2. Monitor Progress: bash watch openstack coe cluster list

    • Status sequence: CREATE_IN_PROGRESS -> CREATE_COMPLETE.
    • If it fails: Check openstack coe cluster show k8s-cluster for errors (usually running out of RAM in the lab environment).

Part 3: Connecting Client to Cluster

Once the status is CREATE_COMPLETE:

  1. Setup Directory: bash mkdir -p ~/.kube

  2. Download Credentials: Magnum generates the TLS certificates needed to talk to the API securely. bash openstack coe cluster config k8s-cluster > ~/.kube/config

  3. Load Config: bash export KUBECONFIG=~/.kube/config

  4. Test Connection: Now we switch from openstack commands to kubectl commands. bash kubectl get nodes Success: You should see 3 nodes (1 master, 2 workers) listed as Ready.

Part 4: Deploying a Workload (Pods)

We have a platform. Let's run code. We will deploy the Nginx app from the lecture notes.

  1. Create Manifest:
nano nginx-app.yaml

apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-web
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web
  template:
    metadata:
      labels:
        app: web
    spec:
      containers:
      - name: nginx
        image: docker.io/nginx
        ports:
        - containerPort: 80
---
apiVersion: v1
kind: Service
metadata:
  name: nginx-lb
spec:
  type: LoadBalancer
  selector:
    app: web
  ports:
  - port: 80

  1. Apply: bash kubectl apply -f nginx-app.yaml

  2. Verify Pods: bash kubectl get pods Reflect: Notice how we have 3 copies running? If we lose a node, Kubernetes will restart them on the other node.

  3. Get the Load Balancer IP: The Service talks to OpenStack Neutron to get a real IP. bash kubectl get svc nginx-lb Look at the EXTERNAL-IP column.

  4. Test: Curl or open that IP in your browser.

Lab Checkpoint

  • I have created a Cluster Template in Magnum.
  • I have successfully provisioned a Kubernetes Cluster (Status: CREATE_COMPLETE).
  • I have configured kubectl and verified node status.
  • I have deployed an Nginx application with 3 replicas.

Reflection:

  • In Lab 3 (Heat), we defined VMs. In Lab 4 (Magnum), we defined Pods. Why is the "Pod" abstraction better for application developers?
  • Who manages the OS updates for the actual nodes in this setup? (Hint: In managed Kubernetes, it's often the Cloud Provider/Magnum; in manual setups, it's you).
