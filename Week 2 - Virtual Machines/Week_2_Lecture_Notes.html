This week focuses on creating and managing virtual machines in Proxmox VE. Students will learn how to properly allocate resources, install guest operating systems, and utilize essential VM management features. By the end of this week, students should be comfortable creating, configuring, and managing VMs in a production-like environment.

Learning Objectives

By the end of this week, students will be able to:

  1. Create and configure virtual machines in Proxmox VE

  3. Install guest operating systems using ISO images
  4. Utilize VM management features such as cloning, snapshots, and console access
  5. Troubleshoot common VM issues and optimize performance

1. Deep Dive: Linux Virtualization (KVM & QEMU)

1.1 The Engine: KVM (Kernel-based Virtual Machine)

  • What is it?: A kernel module that turns Linux into a Type-1 Hypervisor.
  • Function: Handles CPU and Memory virtualization using hardware extensions (VT-x/AMD-V).
  • Performance: Near-native speed because the vCPU runs directly on the physical core.

1.2 The Hardware Emulator: QEMU (Quick Emulator)

  • What is it?: A user-space process that emulates hardware devices.
  • Function: Provides the virtual disk controller, network card, USB ports, and monitor to the VM.
  • Relationship: KVM provides the power (CPU/RAM), QEMU provides the machinery (I/O Devices).

1.3 The Platform: Proxmox VE

  • Role: The management platform that orchestrates KVM and QEMU.
  • It provides the Web UI, API, Clustering, and Backup tools.
  • Analogy:
    • KVM = Engine
    • QEMU = Chassis/Wheels
    • Proxmox = Dashboard/Steering Wheel

2. Strategic Resource Allocation

Sizing Strategies

Instead of generic rules, apply these sizing strategies based on workload type:

Workload Type vCPU Strategy RAM Strategy Disk Strategy
Domain Controller 2 vCPUs (Low usage) 4GB (Cached queries) 60GB (Logs grow)
Database (SQL) 4-8 vCPUs (Parallel queries) 16GB+ (Buffer pool) RAW Block Device (IOPS)
Web Server (Nginx) 2 vCPUs (Event-driven) 2GB (Stateless) QCOW2 (Snapshots)
File Server 2 vCPUs 4GB (ZFS ARC cache) Large, Slow HDD
### The Golden Rules
1. CPU: Never allocate more vCPUs than physical cores. (Avoid Ready Time/Contention).
2. RAM: Never over-commit physical RAM. (Avoid Swap Death).
3. Host: Always reserve 2GB+ RAM for Proxmox itself.

CPU Allocation

Guidelines:

  1. Start Conservative: Begin with 1-2 vCPUs per VM
  2. Monitor Usage: Use top or htop in the guest to check actual CPU usage
  3. Avoid Overcommitment: Total vCPUs across all VMs should not exceed 2-3x physical cores
  4. Consider Workload:
  5. Web servers: 2-4 vCPUs
  6. Databases: 4-8 vCPUs
  7. File servers: 1-2 vCPUs

CPU Types in Proxmox:

  • host: Passes through all host CPU features (best performance, less portable)
  • kvm64: Generic 64-bit CPU (most compatible)
  • Specific models: For compatibility with specific applications

Memory (RAM) Allocation

Guidelines:

  1. Minimum Requirements:
  2. Ubuntu Server: 1 GB (2 GB recommended)
  3. Windows Server: 2 GB minimum (4 GB recommended)

  4. CentOS/Rocky: 1 GB (2 GB recommended)

  5. Working Allocations:

  6. Development VM: 2-4 GB
  7. Web server: 2-4 GB
  8. Database server: 4-8 GB

  9. Desktop VM: 4-8 GB

  10. Ballooning:

  11. Proxmox supports memory ballooning (requires guest agent)
  12. Allows dynamic memory adjustment
  13. Not always reliable; prefer fixed allocations

Memory Overcommitment:

  • Safe: Total VM RAM = 80% of physical RAM
  • Moderate: Total VM RAM = 100-120% of physical RAM
  • Risky: Total VM RAM > 150% of physical RAM

Disk Space Allocation

Considerations:

  1. Thin Provisioning vs. Thick Provisioning:
  2. Thin: Allocates space as needed (saves initial space)

  3. Thick: Pre-allocates full disk space (better performance)

  4. Disk Size Guidelines:

  5. Minimal Linux server: 10-20 GB
  6. Full Linux desktop: 30-50 GB
  7. Windows Server: 40-60 GB

  8. Windows Desktop: 60-100 GB

  9. Storage Backend:

  10. LVM: Direct block access, good performance
  11. LVM-thin: Thin provisioning, snapshots supported
  12. Directory: File-based, flexible but slower
  13. ZFS: Advanced features, high RAM usage

3. VM Creation Process in Proxmox VE

Step-by-Step Guide

Step 1: Prepare ISO Images

  1. Download desired OS ISO (Ubuntu Server, Windows Server, etc.)
  2. Upload to Proxmox via web interface or command line: bash cd /var/lib/vz/template/iso/ wget https://releases.ubuntu.com/22.04/ubuntu-22.04-live-server-amd64.iso

Step 2: Create New VM

  1. Click "Create VM" in Proxmox web interface
  2. General Tab:
  3. Node: Select Proxmox node
  4. VM ID: Unique identifier (100-999999)
  5. Name: Descriptive name (e.g., "ubuntu-web-server")

Step 3: OS Configuration

  1. ISO Image: Select uploaded ISO
  2. Guest OS Type: Linux (2.6/3.x/4.x Kernel) or Windows
  3. SCSI Controller: VirtIO SCSI (best performance)

Step 4: System Settings

  1. BIOS: SeaBIOS (default) or OVMF (UEFI)
  2. Machine: Default (i440fx) or q35 (newer chipset)
  3. SCSI Controller: VirtIO SCSI single

Step 5: Disk Configuration

  1. Bus/Device: SCSI or VirtIO Block
  2. Storage: Select storage backend
  3. Disk Size: Allocate appropriate size
  4. Cache: Write back (default)
  5. Discard: Enable for SSD trim support

Step 6: CPU Configuration

  1. Sockets: Number of CPU sockets (usually 1)
  2. Cores: Number of cores per socket
  3. Type: host or kvm64

Step 7: Memory

  1. Memory (MiB): Allocate RAM in MB
  2. Minimum Memory: For ballooning (optional)

Step 8: Network

  1. Bridge: vmbr0 (default)
  2. VLAN Tag: Optional for network segmentation
  3. Model: VirtIO (paravirtualized)
  4. Firewall: Enable if needed

Step 9: Confirm and Create

  1. Review all settings
  2. Check "Start after created" if desired
  3. Click Finish

5. QEMU Guest Agent

What is QEMU Guest Agent?

The QEMU Guest Agent is a service that runs inside the guest operating system to provide better integration with the hypervisor.

Key Features:

1. Graceful Shutdown/Reboot

  • Allows proper shutdown instead of power-off
  • Preserves file system integrity
  • Closes applications cleanly

2. Resource Reporting

  • Reports actual IP addresses to Proxmox
  • Shows logged-in users
  • Provides more accurate resource metrics

3. Time Synchronization

  • Keeps guest time in sync with host
  • Critical for time-sensitive applications

4. File System Freeze

  • Freezes file system for consistent snapshots
  • Ensures data integrity during backups
  • Thaws automatically after snapshot

Installation

Linux (Debian/Ubuntu):

sudo apt install qemu-guest-agent
sudo systemctl enable qemu-guest-agent
sudo systemctl start qemu-guest-agent

Linux (RHEL/CentOS/Rocky):

sudo yum install qemu-guest-agent
sudo systemctl enable qemu-guest-agent
sudo systemctl start qemu-guest-agent

Windows:

  1. Attach VirtIO driver ISO to VM
  2. Run guest-agent/qemu-ga-x86_64.msi
  3. Follow installation wizard
  4. Service starts automatically

Enabling in Proxmox:

  1. VM Options → QEMU Guest Agent
  2. Enable the checkbox
  3. Restart VM for changes to take effect

6. Proxmox VM Management Features

1. Cloning

Purpose: Create identical copies of VMs quickly

Full Clone:

  • Complete independent copy
  • Takes more time and space
  • Safe for production use

Linked Clone:

  • References base disk, only stores differences
  • Fast and space-efficient
  • Requires base VM to remain available

How to Clone: 1. Right-click VM → Clone 2. Select Full Clone or Linked Clone 3. Choose new VM ID and name 4. Click Clone

Best Practices: - Create a template first, then clone from template - Use linked clones for testing/development - Use full clones for production deployments

2. Snapshots

Purpose: Save VM state at a point in time for easy rollback

Features:

  • Captures disk state and optionally RAM
  • Multiple snapshots per VM supported
  • Quick creation and restoration

How to Create Snapshot: 1. VM → Snapshots tab 2. Click Take Snapshot 3. Enter name and description 4. Choose whether to include RAM 5. Click Take Snapshot

How to Restore Snapshot: 1. Select snapshot from list 2. Click Rollback 3. Confirm action

Important Notes: - Each snapshot creates a new disk layer - Too many snapshots can impact performance - Delete old snapshots after verification - RAM snapshots increase disk usage significantly

3. Console Access

Proxmox provides multiple console options:

noVNC (Web-based):

  • Browser-based access
  • No client installation needed
  • Moderate performance
  • Clipboard sharing supported

SPICE:

  • Better performance than VNC
  • Requires SPICE client (virt-viewer)
  • Clipboard and USB redirection
  • Multi-monitor support

Serial Console:

  • Text-based access
  • Useful for troubleshooting
  • No graphics required

Accessing Console: 1. Select VM 2. Click Console button 3. Choose console type from dropdown

4. Live Migration

Purpose: Move running VM to another Proxmox node without downtime

Requirements: - Proxmox cluster configured - Shared storage (NFS, Ceph, iSCSI) - Same CPU family on both nodes - Network connectivity between nodes

Process: 1. Right-click VM → Migrate 2. Select target node 3. Choose Online migration 4. Click Migrate

Note: Live migration is covered in detail in Week 6 (Clustering)

7. VM vs. Container Comparison

Virtual Machines

Advantages: - Complete Isolation: Full OS with own kernel - Compatibility: Run any OS (Windows, Linux, BSD) - Security: Strong isolation boundary - Legacy Support: Run older applications

Disadvantages: - Resource Overhead: Requires full OS - Slower Boot: Minutes to start - Higher Memory Usage: OS overhead per VM - Larger Disk Space: Full OS installation

Containers (LXC)

Advantages: - Lightweight: Shares host kernel - Fast Boot: Seconds to start - Lower Overhead: Minimal resource usage - High Density: More containers per host

Disadvantages: - Linux Only: Cannot run Windows or BSD - Kernel Dependency: Shares host kernel - Less Isolation: Not as secure as VMs - Compatibility: Some applications may not work

Use Case Guidelines

Choose VMs when: - Running different OS types (Windows, BSD) - Need strong security isolation - Running kernel modules or custom kernels - Legacy application support required

Choose Containers when: - Running Linux-only workloads - Need high density and efficiency - Development and testing environments - Microservices architecture

8. Resource Management Best Practices

1. Start Conservative

  • Begin with minimal resource allocation
  • Monitor actual usage over time
  • Scale up as needed based on metrics

2. Monitor Regularly

  • Use Proxmox built-in monitoring
  • Install monitoring tools (Prometheus, Grafana)
  • Check CPU, memory, disk I/O, and network usage

3. Adjust Dynamically

  • Add resources when bottlenecks are identified
  • Remove excess allocation to improve overall density
  • Balance resources across all VMs

4. Use Snapshots Before Changes

  • Take snapshot before major configuration changes
  • Test changes in cloned environment first
  • Keep recent snapshots for quick rollback

5. Regular Backups

  • Schedule automated backups (weekly minimum)
  • Test restore procedures regularly
  • Store backups on separate storage
  • Keep multiple backup generations

6. Documentation

  • Document VM purposes and configurations
  • Keep inventory of all VMs
  • Track resource allocation changes
  • Document network configurations

9. Common Issues and Troubleshooting

Issue 1: VM Won't Start

Symptoms: VM fails to start or immediately crashes

Possible Causes: - Insufficient host resources - Corrupted disk image - Misconfigured boot order - Hardware incompatibility

Solutions:

# Check host resources
free -h
df -h

# Check VM configuration
qm config <VMID>

# Start VM in debug mode
qm start <VMID> --verbose

# Check system logs
journalctl -u pve-guests.service

Issue 2: Poor VM Performance

Symptoms: Slow response, high latency, low throughput

Possible Causes: - CPU overcommitment - Memory shortage - Disk I/O bottleneck - Network saturation

Solutions: 1. Check resource usage in guest OS 2. Monitor Proxmox host resources 3. Adjust CPU/RAM allocation 4. Change disk cache mode 5. Switch to VirtIO drivers 6. Enable CPU pinning for critical VMs

Issue 3: Network Connectivity Issues

Symptoms: No network connection, slow speeds

Possible Causes: - Incorrect bridge configuration - Missing VirtIO drivers - Firewall blocking traffic - VLAN misconfiguration

Solutions:

# Check bridge configuration
brctl show

# Verify VM network settings
qm config <VMID> | grep net

# Test from host
ping <VM_IP>

# Check firewall rules
iptables -L -n

Issue 4: Disk Space Issues

Symptoms: Disk full errors, cannot write files

Possible Causes: - Thin-provisioned disk overgrowth - Too many snapshots - Large log files - Application data growth

Solutions:

# Check disk usage in guest
df -h

# Check disk images on host
lvs
pvs

# Remove old snapshots
qm listsnapshot <VMID>
qm delsnapshot <VMID> <snapshot_name>

# Resize disk if needed
qm resize <VMID> scsi0 +10G
