Week 8: Compute (Nova) - Lecture Notes

Course: Virtualization and Cloud Infrastructure
Module: OPS3

1. Defining "Compute" in the Cloud

  • The Abstraction: Users do not care about servers; they care about "Instances". Nova provides this abstraction.
  • The Goal: To provide on-demand, scalable computing power without manual intervention.
  • The Scope: Nova manages the lifecycle of the VM. It does not manage the network (Neutron) or the persistent disk (Cinder), but it orchestrates them.

2. Nova Architecture Deep Dive

2.1 The "Shared Nothing" Model

  • Nova follows a "Shared Nothing" architecture (mostly).
  • Scalability: You can add 1,000 Compute nodes without restarting the Controller.
  • Independence: Compute nodes function (mostly) autonomously once they receive a workload. If the API goes down, running VMs keep running.

2.2 The Flow of a Request

  1. Auth: User -> Keystone (Token).
  2. Request: User -> Nova-API (I want a VM).
  3. Queue: Nova-API -> RabbitMQ (Scheduler, help!).
  4. Decision: Nova-Scheduler -> RabbitMQ (Host 'Compute-05' is chosen).

  5. Action: Nova-Compute (on 05) gets the message.

    • Talks to Glance (Get Image).
    • Talks to Neutron (Get Port).
    • Talks to Cinder (Attach Volume - Optional).
    • Talks to Libvirt (Start KVM).

3. The Scheduler: The Intelligent Heart

The Scheduler is the difference between a "Script" and a "Cloud".

3.1 Filters (Can it fit?)

  • Resource Filters: RamFilter, DiskFilter, CoreFilter.
  • Hardware Functionality: PciPassthroughFilter (Does it have a GPU?), NumatopologyFilter (CPU architecture).
  • Affinity: ServerGroupAffinity (Keep these VMs together for speed) vs Anti-Affinity (Keep them apart for redundancy).

3.2 Weighers (Where should it go?)

  • Packing vs Spreading:
    • Packing: Reduces fragmentation, allows powering off unused racks (Green IT).
    • Spreading: Maximizes burst performance for individual VMs, reduces "Noisy Neighbor" effect.

4. Virtualization Driver (Libvirt)

Nova supports many backends, but KVM/Libvirt is the standard.

  • Libvirt: An API library for interacting with Hypervisors.
  • Domain XML: Nova builds a standard XML file describing the VM (CPUs, Memory, Devices) and feeds it to Libvirt.
  • Console Access: Nova starts a "VNC Proxy" so you can view the screen via a Web Browser (Horizon).

5. Security: The Conductor

  • Concept: Hardening the Control Plane.
  • Threat Model: An attacker breaks out of a VM and gains root on the Compute Node.
  • Mitigation: The Compute Node does NOT have the database password. It cannot delete the "Admin" user or change quotas. It must ask the Conductor service (which lives on the protected Controller) to make changes.

6. Summary

  • Nova is a distributed microservice application.
  • It relies heavily on Message Queues (RabbitMQ) for asynchronous communication.
  • The Scheduler is the logic that turns a cluster of servers into a single resource pool.
  • Cells allow Nova to scale to massive sizes.
