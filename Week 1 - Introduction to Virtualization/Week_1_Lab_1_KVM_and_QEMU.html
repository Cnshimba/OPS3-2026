Week 1 - Lab 1: Infrastructure Audit & Virtualization Readiness

Module: Operating Systems 3 (Virtualisation & Cloud Technologies)

Topic: Hardware Virtualization Support Audit
Estimated Time: 30 Minutes

Lab Overview

Before we can build a cloud, we must ensure our hardware can support it. In this lab, you will perform a suitability audit on your computer. You will investigate the CPU architecture, verify the presence of hardware-assisted virtualization flags, and inspect the Linux kernel's hypervisor modules.

Objectives:

  1. CPU Audit: Identify the processor architecture and physical vs logical core count.
  2. Virtualization Verification: Confirm the presence of VT-x (Intel) or AMD-V (AMD) flags.
  3. Kernel Inspection: Verify that the KVM kernel modules are loaded and active.
  4. Memory Analysis: Assess if the system has sufficient RAM for nested virtualization.

Prerequisites: - Access to a Linux environment (Ubuntu Live USB, SystemRescue console, or existing Linux install). - Note: This lab is strictly investigative. You will not be creating VMs yet.

Part 1: CPU Architecture Audit

Understanding your physical hardware is the first step of a Site Reliability Engineer (SRE).

Step 1: Detailed CPU Info

  1. Open a terminal.
  2. Run the following command to get a summary of the CPU: bash lscpu

  3. Field Report: Record the following details from your output:

    • Architecture: ___ (e.g., x8664)
    • Model name: ________
    • Virtualization Flag: [ ] VT-x [ ] AMD-V [ ] None
    • Logical CPUs: ________

Step 2: The "Grep" Check

Automation scripts often need to check this quickly without reading a table.

  1. Run this command to count how many processors support virtualization: bash grep -c -E '(vmx|svm)' /proc/cpuinfo
*   **svm**: Secure Virtual Machine (AMD).
*   **vmx**: Virtual Machine Extensions (Intel).

Part 2: Kernel & Hypervisor Inspection

Linux uses Kernel Modules to add functionality on demand. KVM is just a module that sits inside the Linux kernel.

Step 1: Check for KVM Modules

  1. List all loaded modules and filter for 'kvm': bash lsmod | grep kvm

  2. Field Report:

    • I see the core kvm module.
    • I see the driver: [ ] kvm_intel [ ] kvm_amd
    • Troubleshooting: If you see the core module but NOT the driver, what might be disabled in the BIOS? ________

Step 2: Check KVM Permissions

The hypervisor is exposed to the system as a character device at /dev/kvm. 1. Check the file details: bash ls -l /dev/kvm 2. Investigation:

*   **Owner User**: ____________________
*   **Owner Group**: ____________________
*   **Critical Question**: If your current user is NOT in the group listed above, what error will you get when trying to launch a VM?
    ______________________________________________________________________

Part 3: Resource Suitability Check

Virtualization is resource-hungry. We need to verify we have enough "Headroom" to launch a Cloud.

Step 1: Memory Inspection

  1. Check available RAM: bash free -h

  2. Field Report:

    • Total RAM: ________
    • Available RAM: ________
    • Suitability Verdict: [ ] Ready for Cloud (>16GB) [ ] Minimum Viable (8-16GB) [ ] Upgrade Needed (<8GB)

Step 2: Disk Space

  1. Check partition usage: bash df -h /

  2. Analyze:

    • Free Space: ________
    • Can we store a 50GB Windows Image? [ ] Yes [ ] No

Lab Checkpoint

  • I have confirmed the CPU supports vmx or svm.
  • I have verified that kvm_intel or kvm_amd is loaded.
  • I have determined my total available RAM and Disk space.
  • I understand that /dev/kvm is the bridge between software and hardware.
